# How to write documentation

Documentation exists to help someone understand and use the system. The reader is either a human or a coding agent. Both need the same thing: accurate, clear descriptions of what the system does and how it works, written in plain language.

The most common failure mode is documentation that reads like it was generated by an AI. This happens when the prose is technically correct but empty of substance: full of stock phrases, vague claims, and structure that describes itself. The guidance below addresses this directly.

## Principles

### Accuracy over completeness

A short, correct document is better than a long, slightly-wrong one. If you're not sure about something, leave it out rather than guessing. Wrong documentation is worse than no documentation.

### State facts, don't sell

Documentation describes what the system does. It does not argue that the system is good, important, or innovative. If the system is good, the facts show it. If the facts don't show it, adjectives won't help.

### Write for someone who needs to do something

Every paragraph should help the reader accomplish a task, understand a concept, or make a decision. If a paragraph doesn't do any of these, it's filler.

### Plain language

Use the simplest word that carries the meaning. "Use" not "utilize." "Start" not "commence." "Help" not "facilitate." Formal language creates distance between the reader and the content.

### Show, don't describe

A command someone can run, a file path they can check, a concrete example they can follow. These stick. Abstract descriptions of what something "enables" or "provides" slide off.

## Levels of documentation

Documentation exists at multiple levels. Each has a different audience and purpose.

### Function level

Docstrings and inline comments. Audience: the next developer to touch this code. Purpose: explain non-obvious behavior, invariants, and reasoning. Skip when the code speaks for itself.

### File level

File-header comments and top-of-file docstrings. Audience: a developer opening the file for the first time. Purpose: describe what the file is, what it exports, and any non-obvious dependencies. Skip when the file is small enough that its role is self-evident.

### Module level

README or module-header comments. Audience: someone who wants to understand or use this module. Purpose: describe what the module does, what interfaces it exposes, and how it fits with siblings.

### Codebase level

Top-level README, architecture docs, contributing guides. Audience: new contributors, operators, and future maintainers. Purpose: describe the system's structure, how to build it, how the pieces relate.

### User-facing

Product documentation, help articles, tutorials, API references. Audience: people who use the software from the outside. Purpose: explain how to accomplish tasks.

Each level has its own conventions. The principles above apply to all of them.

## Substance

The most load-bearing section. A document can follow every structural rule below and still feel AI-generated if the substance is thin.

### The substance test

After each paragraph, ask: what does a reader who already knows this area learn from this paragraph? If the answer is "nothing new," the paragraph is filler. Cut it or add something specific.

### State, don't pitch

Documentation is not a pitch. These patterns are pitch voice:

- "In today's..." / "In an era of..." / "Now more than ever..." — delete, state the fact directly
- "It's crucial to understand that..." / "It's worth noting that..." — delete, say the thing
- "A truly elegant solution" / "remarkably powerful" — cut the amplifier, describe what it does
- Adjective stacks: "robust, scalable, enterprise-grade" — pick one specific attribute and show evidence for it

### Be specific

Vague claims add no information. Replace categories with specifics:

```
vague:    significant performance improvement
specific: p99 latency dropped from 180ms to 45ms

vague:    the system handles errors gracefully
specific: when the API returns 429, the client waits 5 minutes
          and retries

vague:    supports multiple deployment targets
specific: runs locally on macOS and Linux, or on AWS Fargate
```

### Show the tradeoff

When documenting a decision, describe what was given up. A decision described only by its benefits reads like marketing. A decision that names what it costs reads like engineering.

```
marketing: We chose S3 for reliable workspace transfer.
engineering: We chose S3 for workspace transfer. ECS Exec piping was
             attempted but Session Manager doesn't handle binary
             data reliably.
```

## Structure

### Lead with what the reader needs

Put the most important information first. If someone is reading docs to figure out how to do something, the command or procedure comes before the explanation of why it works.

### One idea per section

Each section covers one thing. If a section heading needs "and" in it, it's probably two sections.

### Use examples

For any concept that could be misunderstood, include a concrete example. A single example does more than a paragraph of explanation.

### Tables for structured information

When presenting options, comparisons, or reference data, use tables. They're easier to scan than prose and harder to misread.

### Code blocks for commands and file contents

Anything the reader might copy or reference should be in a code block. File paths, commands, status values, configuration — code blocks signal "this is exact."

### Prefer single-line paragraphs

Prefer single-line paragraphs over hard-wrapping. Agent read tools return lines by line range 1-N — single-line paragraphs let one read pull in more distinct paragraphs. Exceptions: hard-wrap when the project convention requires it, or when a paragraph runs longer than ~5 sentences and becomes hard to scan on one line. Usually the better fix for a long paragraph is to split it into shorter ones.

### Contents TOC in reference files

Markdown reference files — long-form guides and expertise files typically read by agents in parts — should open with a `## Contents` nested bulleted list matching the file's H2/H3 structure (like this file does). Don't add one to procedural markdown files (SKILL.md, prompts) or files with fewer than three top-level sections.

## Vocabulary consistency

Documentation reads as one system when it uses one term per concept. When "user" and "member" appear on different pages for the same thing, or "batch job" in the CLI is called "scheduled task" in the docs, readers lose trust and search stops working. When editing or reviewing, note terms that appear in more than one form and pick a canonical version.

## AI writing tells to avoid

AI-generated documentation has distinctive patterns that readers and agents recognize. Even if the content is correct, these patterns cause readers to distrust the document. The goal is perception, not just correctness.

### Words to avoid

These words in their figurative senses are strong AI signals. Use plain alternatives:

**Strongest signals:** `delve`, `tapestry`, `realm`, `landscape` (metaphorical), `navigate` (metaphorical), `leverage` (verb), `utilize`, `harness`, `robust`, `seamless`, `pivotal`, `crucial`, `comprehensive`, `multifaceted`, `holistic`, `nuanced`, `paradigm`, `synergy`, `meticulously`, `vibrant`, `underscore` (figurative), `foster`, `cultivate`, `showcasing`, `highlighting`, `emphasizing`, `enhance`, `align with`, `bolster`

**Use with care:** `transformative`, `innovative`, `groundbreaking`, `game-changer`, `elevate`, `empower`, `streamline`, `optimize`, `spearhead`, `reimagine`, `redefine`, `deep dive`, `at its core`

**Prefer the plain word:**

| Instead of | Use |
|---|---|
| utilize | use |
| facilitate | help, make possible |
| commence | start |
| leverage | use |
| endeavor | try |
| plethora | many |
| myriad | many |
| elucidate | explain |

### Phrases to avoid

**Stock openings — never start a section with:**
- "In today's..." / "In the ever-evolving landscape of..."
- "As we navigate..." / "As AI continues to..."
- "Let's dive in" / "Let's explore..."

**Stock transitions — cut or replace:**
- "It's important to note that" — say the thing
- "It's worth mentioning that" — say the thing
- "Furthermore" / "Moreover" / "Additionally" — use "also" or restructure
- "In conclusion" / "In summary" — just conclude
- "As mentioned earlier" — just state it again

**Stock closings:**
- "The future of X is bright"
- "Only time will tell"
- "One thing is certain"

### Sentence patterns to avoid

**"Not just X, it's Y"** — The most distinctive AI rhetorical pattern. Variants: "It's not just X, it's Y," "More than just X; it's Z," "Not only X, but also Y." If a document has three or more of these, it reads as AI. Use at most one, and only when the contrast is genuinely load-bearing.

**Self-posed questions answered immediately** — "So what is X? Well, X is..." is an AI stock move. Either ask a real question and let the reader think, or just state the fact.

**Meta-commentary about the document** — "In this section, we will explore..." / "Let's now turn our attention to..." If a sentence describes what the document is about to do, cut it and do the thing.

**Avoidance of plain "is"** — AI substitutes "serves as," "features," "offers" for the plain "is." "The system is a workflow engine" is better than "The system serves as a workflow engine."

**Rule of three as default** — AI puts everything in threes. Vary the count. Use two items. Use four. Use one.

### Formatting tells

- **Random bolding** — only bold things that genuinely need emphasis. Bold should be rare.
- **Inline-header lists** — the `**Heading:** description` pattern repeated down a page is a direct AI signature. Promote to sub-sections (H3/H4), convert to prose, or use plain lists.
- **Formulaic subheadings** — "Understanding X," "The Importance of Y," "Key Takeaways." Use descriptive, specific headings.
- **Uniform paragraph length** — vary paragraph length deliberately.
- **Emoji in headings** — never.

## Editing checklist

When writing or reviewing documentation:

1. **Accuracy** — does every factual claim match the code?
2. **Substance** — does each paragraph teach the reader something? If not, cut it or add specifics.
3. **Pitch voice** — scan for stock openers, amplifiers, adjective stacks, meta-commentary. Cut each one.
4. **AI tells** — scan for the word list above. Replace with plain alternatives.
5. **Specificity** — does each section have at least one concrete example, command, file path, or number?
6. **Plain language** — any word of four syllables that has a one-syllable equivalent? Use the simpler word.
7. **The sniff test** — read each paragraph and ask: does this sound like a person wrote it, or an AI? If AI, diagnose: pitch voice (cut the selling), missing substance (add specifics), or AI tells at the word level (replace the words).

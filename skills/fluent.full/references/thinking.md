# Thinking frameworks

## Default mode

The Socratic method — asking questions rather than making assertions — is the baseline mode of engagement across all these frameworks. Reach for a specific framework below when a particular tell surfaces.

## When to use which framework

| Goal | Frameworks |
|------|-----------|
| Default mode of engagement | Socratic Method |
| Slow intuitive reasoning to deliberate thinking | System 1 vs. 2 |
| Probe for information the user hasn't checked | WYSIATI |
| Ground optimism in reality | Outside View |
| Understand the real outcome being sought | Jobs to Be Done |
| Drill past symptoms to root cause | 5 Whys, First Principles |
| Explore the problem deeply, then explore solutions before converging | Design Thinking |
| Challenge assumptions | First Principles, Socratic Method, Inversion |
| Probe which parts of the solution are load-bearing | SCAMPER |
| Explore multiple angles one at a time | Six Thinking Hats |
| Find failure modes | Pre-Mortem, Inversion |
| Stress-test an argument | Toulmin, Steelmanning |
| Sharpen a position against the strongest counter | Steelmanning |
| Arrive at a more complete view than either position alone | Dialectic |
| Decide if debate is worth having | RISA |
| Present an idea clearly | PEEL, Toulmin |

## Frameworks

### System 1 vs. System 2

When the user asserts something as obvious or reaches a conclusion without visible reasoning, that's usually intuition rather than deliberate thinking. Slow the conversation down and ask them to walk through the reasoning — especially when the conversation is about a novel idea or a high-stakes decision.

### WYSIATI — What You See Is All There Is

When the user is confident about something they don't have full information on, they may have substituted assumptions for facts without noticing. Probe the assumptions ("what would we need to know to check this?" / "what are we taking for granted?"), and where the missing information is fetchable, do the research — read the codebase, search online, or consult external systems.

### Outside View / Reference Class Forecasting

When the user is assessing whether an idea or approach will work by reasoning from its specifics — arguing that this one is different, or that a familiar comparison doesn't apply — ask what category of thing it is and how similar ideas in that class have actually turned out. Where the base rate isn't obvious, do the research — read the codebase for similar work, search online, or consult external systems. Anchor on that base rate, then adjust modestly for genuinely distinguishing features. The base rate is usually more predictive than the specific story.

### Jobs to Be Done

When the user proposes a solution before they've named what someone is actually trying to accomplish, reframe by asking what "job" the idea would be hired to do. Probe for functional (the practical task), emotional (how it makes them feel), and social (how it affects how others see them) dimensions. Concrete openers: "when does the urge to have this arise?" / "what triggers it?" / "what would they do instead if this didn't exist?"

### 5 Whys

When the user's problem statement sounds like a symptom rather than a cause, chain "why?" until you hit an actionable root cause. Ground each answer in evidence rather than guessing; follow multiple branches when the first doesn't feel like bedrock. Stop when the next "why?" has no useful answer.

### Design Thinking (Double Diamond)

When the user is treating the problem as settled — the brief takes it as given, the solution shape is already baked into the framing, or the underlying need hasn't actually been explored — walk the double diamond: **Discover** (explore the problem space broadly — resist defining the problem yet), **Define** (converge on a precise problem statement based on what you found), **Develop** (generate multiple possible solutions without filtering), **Deliver** (converge on the best option, prototype, refine). Most ideas that fail are solving the wrong problem — don't leave the first diamond until the problem itself is understood.

### First Principles Thinking

When the user reaches for a familiar approach because it's how things are done, walk through the embedded assumptions with them. First, ask them to explain the approach from scratch as if to a child — this strips jargon and surfaces the assumptions. Then for each assumption, ask "how do we really know this is true?" and "what if this weren't true?" to see what alternative shapes exist. Rebuild using only what survives.

### Socratic Method

The default mode when helping the user think through an idea: ask questions rather than making assertions. Draw from a set of question types — clarification ("what do you mean by X?"), assumptions ("what are we taking for granted?"), evidence ("what makes us confident?"), perspective ("how would someone who disagrees see this?"), and implications ("if this is true, what follows?"). Stay genuinely curious — the goal is helping them examine the idea, not leading them to a conclusion.

### SCAMPER

When the user is stuck on how to make an idea work in its current shape, or defending a specific shape without having tested whether alternatives would work as well, force the shape through SCAMPER's seven transformations: **Substitute** (swap a component), **Combine** (merge with something else), **Adapt** (borrow from elsewhere), **Modify** (change scale, emphasis, or form), **Put to other uses**, **Eliminate** (remove a piece — does it still work?), **Reverse/Rearrange** (flip order or structure). For each: does the idea still work if you apply the transformation? The point is exposing which components are load-bearing and which are conventions.

### Six Thinking Hats

When the discussion of an idea would benefit from structure — stuck in one mode too long, cycling through the same two perspectives, or skipping an important angle — walk through the six modes deliberately, one at a time: **White** (facts only — what do we know?), **Yellow** (optimism only — what's the best case?), **Black** (critique only — what could go wrong?), **Red** (gut feeling, no justification), **Green** (creative alternatives — how else could this work?), **Blue** (meta — are we asking the right question?). Complete one mode before moving to the next; use Blue to set the sequence and synthesize what surfaced at the end.

### Pre-Mortem

When the user is close to committing to a plan and wants a final check, project forward: ask "it's a year from now — this failed badly, not just underperformed. What went wrong?" Framing failure as a fact rather than a possibility surfaces concerns that forward planning tends to soften. Each cause becomes a concrete risk to mitigate before commitment.

### Inversion

When the user has a plan or idea they're confident about but hasn't stress-tested, flip the question: instead of "how do we make this succeed?", ask "what would guarantee this fails?" Failure modes are usually more concrete and enumerable than success paths — listing them and designing around what surfaces catches risks that forward planning tends to skip. The same flip works on a claim: instead of arguing why it's true, ask "what would have to be true for it to be false?"

### Toulmin Argument Model

When the user's argument sounds right but isn't landing, map it onto Toulmin's six parts and look for gaps: **Claim** (the assertion), **Grounds** (evidence for it), **Warrant** (the logical bridge from evidence to claim), **Backing** (evidence validating the warrant), **Qualifier** (how strong — always, usually, probably?), **Rebuttal** (when the claim doesn't hold). Most arguments skip the warrant — the evidence and claim get stated, the bridge between them gets assumed. A claim with no qualifier is overreaching; a warrant with no backing is an assumption in disguise.

### Steelmanning

When the user is dismissing an alternative approach, rejecting a critique, or ruling out an option, first help them reconstruct the opposing view in its strongest, most charitable form — add the best supporting evidence, fill gaps generously, and phrase it the way a thoughtful proponent would. Verify: "the strongest version of that is X — is that fair?" Only critique once they agree. After critiquing, check whether the disagreement has narrowed or the strongest version of the opposing argument has something worth incorporating.

### Dialectic Method (Thesis-Antithesis-Synthesis)

When the user is stuck in a binary debate where both positions have real merit, or picking between two options that each have force, run the dialectic: **Thesis** — state the idea as forcefully as possible; **Antithesis** — argue the strongest genuine opposite, not a rebuttal of weak points; **Synthesis** — find the position that resolves the tension between them, often a higher-order view that preserves what's valid in both. Synthesis isn't compromise (splitting the difference) — it's a reframing that makes the original conflict less fundamental. The synthesis question isn't "who's right?" but "what does a view that takes both positions seriously look like?"

### RISA Framework

Before the user spends energy on a disagreement — engaging one, escalating one, or continuing one that keeps recurring — check whether it's worth having: **Real** (an actual disagreement, or a miscommunication or imagined slight?), **Important** (does the outcome matter enough to justify the argument?), **Specific** (concrete enough to make progress, or too vague?), **Aligned** (do both sides want to understand, or is one side just wanting to win or vent?). If any of the four fails, fix that first rather than proceeding.

### PEEL Structure

When the user needs to present a single argument clearly — a claim in a brief, a justification for an approach, a defense of a decision — structure it as PEEL: **Point** (state the claim plainly, one sentence), **Evidence** (the most direct fact or example that supports it, not general support), **Explanation** (spell out the connection from evidence to claim — don't assume it's obvious), **Link** (connect back to the broader question so the reader knows why this matters). Explanation is the step people skip most; making it explicit exposes any gaps in the reasoning.
